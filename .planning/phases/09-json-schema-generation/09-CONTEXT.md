# Phase 9: JSON Schema Generation - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Add `#[derive(JsonSchema)]` on all config types, manual impls for `StringOrArray` and `PullPolicy`, committed `anyclaw.schema.json`, `anyclaw schema` CLI subcommand, and `anyclaw validate` enhanced with JSON Schema validation.

</domain>

<decisions>
## Implementation Decisions

### schemars dependency
- **D-01:** Add `schemars = "1"` to `crates/anyclaw-config/Cargo.toml` (already in workspace via `anyclaw-sdk-tool`)
- **D-02:** Add `jsonschema = "0.28"` as dev-dependency for schema validation tests (research recommended 0.45 but check latest compatible)

### Derive-based types (19 types)
- **D-03:** Add `JsonSchema` to derive list on all 19 types that use `#[derive(Deserialize)]`: `LogFormat`, `SqliteStoreConfig`, `SessionStoreConfig`, `AnyclawConfig`, `BackoffConfig`, `CrashTrackerConfig`, `AgentsManagerConfig`, `ChannelsManagerConfig`, `ToolsManagerConfig`, `WorkspaceConfig`, `AgentConfig`, `ChannelConfig`, `ReactionLifecycle`, `AckConfig`, `ToolType`, `ToolConfig`, `WasmSandboxConfig`, `PreopenedDir`, `SupervisorConfig`
- **D-04:** `LocalWorkspaceConfig` and `DockerWorkspaceConfig` also get derive — the `deserialize_string_map` on `env` fields generates `HashMap<String, String>` schema which is an acceptable approximation

### Manual impls (2 types)
- **D-05:** `StringOrArray` needs manual `impl JsonSchema` — schema is `oneOf [{ type: string }, { type: array, items: { type: string } }]`
- **D-06:** `PullPolicy` needs manual `impl JsonSchema` — schema is `{ type: string, enum: ["always", "never", "if_not_present"], default: "if_not_present" }`

### Committed schema file
- **D-07:** Generate and commit `anyclaw.schema.json` at repository root
- **D-08:** Schema generated by calling `schemars::generate::SchemaSettings::draft2020_12().into_generator().into_root_schema_for::<AnyclawConfig>()` (or equivalent schemars 1.x API)

### CLI subcommand
- **D-09:** Add `anyclaw schema` subcommand to `cli.rs` that prints JSON Schema to stdout
- **D-10:** Implementation: instantiate schema generator, serialize to pretty JSON, print

### Validate enhancement
- **D-11:** `anyclaw validate` enhanced to validate config against JSON Schema before the existing binary path checks
- **D-12:** Use `jsonschema` crate for runtime validation (add as regular dependency, not just dev)

### Agent's Discretion
- Exact schemars 1.x API calls (may differ from 0.x)
- Schema file formatting (2-space indent, trailing newline)
- Error message formatting for schema validation failures
- Whether to add `$schema` and `$id` fields to generated schema

</decisions>

## Existing Code Insights

### Reusable Assets
- `schemars = "1"` already in workspace via `anyclaw-sdk-tool` — version compatible
- `cli.rs` uses Clap derive with subcommand enum — adding `Schema` variant is mechanical
- Existing `validate` subcommand in cli.rs provides the pattern for `schema` subcommand

### Established Patterns
- All config types already derive `Serialize` — schemars 1.x reads serde attributes directly
- `serde(tag = "type", rename_all = "snake_case")` on `SessionStoreConfig` and `WorkspaceConfig` — schemars generates `oneOf` with discriminator
- `serde(rename_all = "lowercase")` on `LogFormat`, `ToolType` — schemars handles this
- `HashMap<String, serde_json::Value>` on options fields — schemars generates `additionalProperties: {}`

### Integration Points
- `crates/anyclaw-config/src/lib.rs` — Schema generation function lives here (next to config loading)
- `crates/anyclaw/src/cli.rs` — `schema` subcommand dispatches to config crate
- `anyclaw.schema.json` at repo root — committed artifact
