<objective>
Combine multiple tool calls into a single Telegram message per turn, with each tool shown as a status line that updates in-place.

Purpose: Reduce message spam when agents make many tool calls. Currently each tool call is a separate message — noisy and hard to follow. A single combined message with status lines (like a task list) is cleaner UX.

Output: Modified turn tracking and deliver logic in the Telegram channel.
</objective>

<task type="auto">
  <name>Task 1: Add combined tool message tracking to ChatTurn</name>
  <files>ext/channels/telegram/src/turn.rs</files>
  <action>
Modify `ToolCallTrack` to include a `status` field tracking the current state of each tool call:

```rust
pub struct ToolCallTrack {
    pub name: String,
    pub status: ToolCallStatus,
}

pub enum ToolCallStatus {
    Started,     // 🔧 (just called, no update yet)
    InProgress,  // ⏳
    Completed,   // ✅
    Failed(Option<String>),  // ❌ (with optional error output)
}
```

Remove `msg_id` from `ToolCallTrack` — individual tool calls no longer own their own message.

Add a `tools_msg_id: i32` field to `ChatTurn` (default `0`) to track the single combined tools message for the turn. This is the Telegram message ID that gets edited as tools are added/updated.

Add a `tool_call_order: Vec<String>` field to `ChatTurn` to preserve insertion order of tool_call_ids (HashMap doesn't guarantee order, but we want tools displayed in the order they were called).

Update `ChatTurn::new()` to initialize `tools_msg_id: 0` and `tool_call_order: Vec::new()`.

Update `ChatTurn::cleanup()` to also clear `tools_msg_id = 0` and `tool_call_order.clear()`.

Add a method `ChatTurn::render_tools_text(&self) -> String` that iterates `tool_call_order`, looks up each in `tool_calls`, and builds the combined HTML text. Each line format:
- `Started`: `🔧 <code>{name}</code>…`
- `InProgress`: `⏳ <code>{name}</code>`
- `Completed`: `✅ <code>{name}</code>`
- `Failed(None)`: `❌ <code>{name}</code>`
- `Failed(Some(err))`: `❌ <code>{name}</code>\n<pre>{err}</pre>`

Use `escape_html` from `crate::formatting` for all name/error values. Lines joined with `\n`.

Update existing tests that reference `ToolCallTrack { msg_id, name }` to use the new struct shape.
  </action>

  </verify>

<task type="auto">
  <name>Task 2: Rewrite ToolCall and ToolCallUpdate handlers to use combined message</name>
  <files>ext/channels/telegram/src/deliver.rs</files>
  <action>
Rewrite the `ContentKind::ToolCall` match arm:

1. Acquire write lock on turns, get-or-create the turn (same pattern as existing code).
2. Insert into `turn.tool_calls` with `ToolCallStatus::Started`. Push `tool_call_id` to `turn.tool_call_order`.
3. Call `turn.render_tools_text()` to get the combined HTML.
4. Read `turn.tools_msg_id`. Drop the lock.
5. If `tools_msg_id == 0` (first tool call): `bot.send_message()` with the combined text, then write lock again and set `turn.tools_msg_id = sent.id.0`.
6. If `tools_msg_id != 0` (subsequent): `bot.edit_message_text()` on the existing message with the updated combined text.

Rewrite the `ContentKind::ToolCallUpdate` match arm:

1. Acquire write lock on turns.
2. Look up the tool_call_id in `turn.tool_calls`. Update its status based on the `status` string:
   - `"in_progress"` → `ToolCallStatus::InProgress`
   - `"completed"` → `ToolCallStatus::Completed`
   - `"failed"` → `ToolCallStatus::Failed(output.clone())`
3. Call `turn.render_tools_text()`. Read `turn.tools_msg_id`. Drop the lock.
4. If `tools_msg_id != 0`: `bot.edit_message_text()` with the updated combined text.
5. If tool_call_id not found in tracking (edge case — update arrived before call), fall back to logging a warning and doing nothing.

Remove the old `format_tool_call_update_text()` function — it's replaced by `ChatTurn::render_tools_text()`. Update or remove its tests accordingly. The test logic for HTML escaping and emoji selection should move to unit tests on `render_tools_text()` in `turn.rs`.

All Telegram API calls must use `retry_telegram_op()` wrapper and `ParseMode::Html`. Follow existing lock discipline: never hold write lock across `.await` points.
